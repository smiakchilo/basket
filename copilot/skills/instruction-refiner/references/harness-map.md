# Harness Map

Reference data for instruction-refiner's harness-agnostic operation.

## Detection

See **SKILL.md Step 1** for the load-path detection table and fallback prompt.

## Target Paths

Where to look for and write customization files, per harness.

### Copilot

Search in this order (least global → most global):

| Scope | Instructions | Skills |
|-------|-------------|--------|
| Project | `.github/copilot-instructions.md`<br>`.github/instructions/*.instructions.md`<br>`.github/*.instructions.md` | `.github/skills/` |
| VS Code user | `%APPDATA%\Code\User\prompts\` | — |
| User global | `~/.copilot/instructions/*.instructions.md` | `~/.copilot/skills/<name>/SKILL.md` |

### Claude

Search in this order:

| Scope | Instructions | Skills |
|-------|-------------|--------|
| Project | `.claude/rules/*.md` | — |
| User global | `~/.claude/rules/*.md` | `~/.claude/skills/<name>/SKILL.md` |
| User global entry | `~/.claude/CLAUDE.md` | — |

### Codex

| Scope | Instructions | Skills |
|-------|-------------|--------|
| Project | `<project>/.codex/AGENTS.md`<br>Subdirectory `AGENTS.md` files (e.g. `core/AGENTS.md`) | `.codex/skills/<name>/SKILL.md` |

Codex has no user-global instruction store. Rules are written directly into the closest `AGENTS.md`.

## Scope Heuristic

Applies to all harnesses:

- Correction is **project-specific** (e.g. AEM patterns, project test conventions) → prefer project-level file.
- Correction is **language/framework-generic** (e.g. Java naming, Mockito style) → prefer user-level file (n/a for Codex).
- Correction reveals a flaw in **how a skill drives a workflow** → update the relevant `SKILL.md` at the harness skill path.
- Correction reveals a flaw in **how an agent is described or triggered** → update the relevant agent definition file.

## File Formats

### Copilot — `*.instructions.md`

```markdown
---
description: "Use when: <context>. Covers <topics>."
applyTo: "**/*.java"
---

# Rule title

Rule body...
```

Key points:
- Frontmatter key for file scope: **`applyTo:`** (glob pattern string, not array)
- Always-on (no path restriction): omit `applyTo` or set to `**`
- File suffix: `.instructions.md`

### Claude — `rules/*.md`

```markdown
---
description: >-
  Use when <context>. Covers <topics>.
paths: '**/*.java'
---

# Rule title

Rule body...
```

Key points:
- Frontmatter key for file scope: **`paths:`** (single quoted string, or array for multi-glob)
- **Never use `applyTo:` in a Claude rules file** — it is a Copilot-only key
- Always-on (no path restriction): omit `paths` entirely
- File suffix: `.md` (not `.instructions.md`)

### Codex — `AGENTS.md`

No YAML frontmatter. Rules are embedded as plain Markdown sections.

```markdown
# Existing AGENTS.md content...

---

## <Rule topic>

Rule body...
```

Key points:
- **Never add YAML frontmatter** to an `AGENTS.md` file
- Insert new rules as top-level `##` sections at an appropriate position (after existing rules of similar scope)
- If the file carries an `<!-- Auto-generated by export.mjs -->` header, note it but edit anyway — the edit will be overwritten by the next `export.mjs` run unless the basket source is also updated (see Codex Authoring Policy below)

## Codex Authoring Policy

Codex `AGENTS.md` files are auto-generated from basket sources via `export.mjs`. Direct edits are overwritten on next install.

Before writing to a Codex `AGENTS.md`:

1. Check whether a basket repo is accessible: look for `export.mjs` at `$(git rev-parse --show-toplevel)/export.mjs` and verify that `copilot/instructions/` exists alongside it.
2. **If basket is accessible**: edit the appropriate source file under `copilot/instructions/` (using Copilot format), then run `node export.mjs` to regenerate all derived outputs. Inform the user that `AGENTS.md` will be updated on the next `npm run export` or `node install.mjs`.
3. **If basket is absent**: edit `AGENTS.md` directly and warn the user that the change is not backed by a source file and will be overwritten if `install.mjs` is run later.

## Safe-Write Guard

Before creating or modifying any customization file, verify the target directory exists:

- PowerShell: `Test-Path "<directory>"`
- Bash: `[ -d "<directory>" ]`

If the directory does not exist, warn the user and stop. Do not create parent directories silently.
